## 如何贡献

### 报告问题

如果你发现以下问题，请通过 [Issues](../../issues) 页面报告：

- 🔗 **链接失效**：数据链接无法访问
- 📊 **信息过时**：数据源信息需要更新
- ❌ **数据错误**：数据描述或分类不正确

### 添加新数据源

添加新数据源时，请确保：

1. **数据可公开访问**：无需注册或付费即可访问
2. **来源可靠**：来自官方或知名机构
3. **格式规范**：按照现有分类和表格格式添加
4. **信息完整**：包含所有必需字段

**数据源添加模板：**
```markdown

| 数据名称 | 提供机构 | 空间/时间分辨率 | 更新频率 | 数据覆盖时段 | [链接](URL) | 许可类型 |
```

## 贡献流程

### 1. Fork 项目仓库

1. 点击 GitHub 页面右上角的 "Fork" 按钮
2. 将项目克隆到本地：
```bash
git clone https://github.com/wait4xx/Meteorological-Data-Download-Guide.git
cd Meteorological-Data-Download-Guide
```

### 2. 创建分支

为你的贡献创建功能分支：
```bash
git checkout -b feature/your-feature-name
```
分支命名约定：
- `feature/data-source-name` - 添加新数据源
- `fix/issue-description` - 修复问题
- `docs/topic` - 文档改进

### 3. 进行修改

在本地进行修改，确保：
- 遵循项目的代码和文档规范
- 测试所有链接的有效性
- 更新相关的文档

### 4. 提交更改

使用描述性的提交信息：
```bash
git add .
git commit -m "feat: 添加中国气象局全球模式数据源"
```

提交信息格式：
- `feat:` 新功能或数据源
- `fix:` 修复问题
- `docs:` 文档更新
- `style:` 格式调整
- `refactor:` 代码重构

### 5. 推送并创建 Pull Request

```bash
git push origin feature/your-feature-name
```

然后在 GitHub 上创建 Pull Request，并填写 PR 模板。

## 数据源标准

### 可接受的数据源

✅ **官方气象机构**：WMO成员机构、国家气象局等
✅ **科研机构**：大学、研究所的公开数据
✅ **政府开放数据**：政府发布的免费气象数据
✅ **国际组织**：联合国、世界银行等机构数据
✅ **社区项目**：有良好维护的开源气象项目

### 不可接受的数据源

❌ **商业数据**：需要付费订阅的数据
❌ **限制性数据**：有严格使用限制的数据
❌ **个人博客**：非官方、未经验证的数据源
❌ **侵权内容**：侵犯版权或协议的数据

### 数据质量要求

- **可访问性**：链接稳定，无需特殊权限
- **时效性**：数据定期更新
- **完整性**：提供元数据和文档
- **准确性**：数据来源可靠准确

## 文档规范

### Markdown 格式

- 使用标准的 Markdown 语法
- 表格对齐使用冒号 `:---`、`:--:`、`---:`
- 链接使用描述性文字 `[描述](URL)`
- 图片添加替代文本 `![描述](image-url)`

### 数据表格字段

| 字段 | 必需 | 描述 | 示例 |
|------|------|------|------|
| 数据源 | 是 | 数据产品或服务的名称 | GFS 全球预报系统 |
| 机构 | 是 | 数据提供机构 | NOAA/NCEP |
| 分辨率 | 是 | 空间和时间分辨率 | 0.25°, 3小时 |
| 更新频率 | 是 | 数据更新频率 | 每日4次 |
| 数据时段 | 否 | 历史数据覆盖范围 | 2000-现在 |
| 访问方式 | 是 | 数据获取链接 | [NOMADS](https://nomads.ncep.noaa.gov/) |
| 许可协议 | 是 | 数据使用许可 | CC BY 4.0 |

## 评审流程

### Pull Request 检查清单

在提交 PR 前，请确认：

- [ ] 代码/文档遵循项目规范
- [ ] 所有链接有效且可访问
- [ ] 添加了必要的测试（如适用）
- [ ] 更新了相关文档
- [ ] 提交信息清晰明确
- [ ] 分支与主分支无冲突

### 评审标准

1. **功能性**：贡献解决了什么问题
2. **质量**：代码/文档质量是否符合标准
3. **测试**：是否包含适当的测试用例
4. **文档**：是否更新了相关文档
5. **兼容性**：是否与现有内容兼容
