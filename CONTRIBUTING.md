## 如何贡献

### 报告问题

如果你发现以下问题，请通过 [Issues](../../issues) 页面报告：

- 🔗 **链接失效**：数据链接无法访问
- 📊 **信息过时**：数据源信息需要更新
- ❌ **数据错误**：数据描述或分类不正确
- 💡 **功能建议**：对项目改进的建议
- ➕ **资源推荐**：推荐新的数据源

**问题报告模板：**
```markdown

## 建议解决方案（可选）
<!-- 如果有解决方案建议，请在此说明 -->

## 环境信息
- 操作系统：
- 浏览器：
- 发现时间：
```

### 添加新数据源

添加新数据源时，请确保：

1. **数据可公开访问**：无需注册或付费即可访问
2. **来源可靠**：来自官方或知名机构
3. **格式规范**：按照现有分类和折叠卡片格式添加
4. **信息完整**：包含所有必需徽章和链接

**数据源添加模板：**

```markdown
<details>
<summary><b>数据名称</b> · 机构/简短描述</summary>

![分辨率](https://img.shields.io/badge/分辨率-0.25°-blue?style=flat-square)
![时效](https://img.shields.io/badge/时效-0~384h(3h)-green?style=flat-square)
![更新](https://img.shields.io/badge/更新-每日4次-orange?style=flat-square)
![来源](https://img.shields.io/badge/来源-数据提供方-颜色代码?style=flat-square)

🔗 [数据链接](https://example.com/data) · 📅 数据时段 · 📝 [示例代码](./sources/example.py)

</details>
```

**徽章颜色规范：**

| 来源类型 | 颜色代码 | 示例 |
|----------|----------|------|
| 官方机构 | `9CF` | CMA, DWD, JMA |
| NOAA | `00CED1` | NOAA, NCEP |
| NASA | `FF0000` | NASA, MERRA-2 |
| UCAR | `800080` | UCAR, NCAR |
| ECMWF/CDS | `00CED1` | ECMWF, CDS |
| ESA/Copernicus | `003399` | ESA, Copernicus |
| AWS | `FF9900` | AWS S3 |
| Google | `4285F4` | Google Cloud |
| OpeNDAP | `00BFFF` | OpeNDAP 服务 |
| 其他 | `00BFFF` | 其他数据提供方 |

**徽章字段说明：**

| 徽章 | 用途 | 适用数据类型 |
|------|------|--------------|
| 分辨率 | 空间分辨率 | 格点数据 |
| 时效 | 预报时效范围 | 预报数据 |
| 更新 | 数据更新频率 | 所有数据 |
| 来源 | 数据提供方 | 所有数据 |
| 时间分辨率 | 时间间隔 | 再分析/观测数据 |
| 时间范围 | 历史数据覆盖 | 历史数据 |
| 魔法 | 需要特殊网络 | 需VPN访问 |
| 类型 | 数据类型 | 气候/观测数据 |

## 贡献流程

### 1. Fork 项目仓库

1. 点击 GitHub 页面右上角的 "Fork" 按钮
2. 将项目克隆到本地：
```bash
git clone https://github.com/wait4xx/Meteorological-Data-Download-Guide.git
cd Meteorological-Data-Download-Guide
```

### 2. 创建分支

为你的贡献创建功能分支：
```bash
git checkout -b feature/your-feature-name
```
分支命名约定：
- `feature/data-source-name` - 添加新数据源
- `fix/issue-description` - 修复问题
- `docs/topic` - 文档改进

### 3. 进行修改

在本地进行修改，确保：
- 遵循项目的代码和文档规范
- 测试所有链接的有效性
- 更新相关的文档

### 4. 提交更改

使用描述性的提交信息：
```bash
git add .
git commit -m "feat: 添加中国气象局全球模式数据源"
```

提交信息格式：
- `feat:` 新功能或数据源
- `fix:` 修复问题
- `docs:` 文档更新
- `style:` 格式调整
- `refactor:` 代码重构

### 5. 推送并创建 Pull Request

```bash
git push origin feature/your-feature-name
```

然后在 GitHub 上创建 Pull Request，并填写 PR 模板。

## 数据源标准

### 可接受的数据源

✅ **官方气象机构**：WMO成员机构、国家气象局等
✅ **科研机构**：大学、研究所的公开数据
✅ **政府开放数据**：政府发布的免费气象数据
✅ **国际组织**：联合国、世界银行等机构数据
✅ **社区项目**：有良好维护的开源气象项目

### 不可接受的数据源

❌ **商业数据**：需要付费订阅的数据
❌ **限制性数据**：有严格使用限制的数据
❌ **个人博客**：非官方、未经验证的数据源
❌ **侵权内容**：侵犯版权或协议的数据

### 数据质量要求

- **可访问性**：链接稳定，无需特殊权限
- **时效性**：数据定期更新
- **完整性**：提供元数据和文档
- **准确性**：数据来源可靠准确

## 文档规范

### Markdown 格式

- 使用标准的 Markdown 语法
- 使用 `<details>` 标签创建可折叠卡片
- 链接使用描述性文字 `🔗 [描述](URL)`
- 图片添加替代文本 `![描述](image-url)`
- 使用 shields.io 徽章展示元数据

### 徽章格式

所有数据源条目应使用徽章展示关键信息：

```markdown
![标签](https://img.shields.io/badge/标签-值-颜色?style=flat-square)
```

**颜色含义：**
- `blue` - 空间分辨率等基础信息
- `green` - 时效、时间分辨率等时间信息
- `orange` - 更新频率、时间范围等
- 自定义颜色 - 数据来源标识

### 数据卡片结构

每个数据源应包含以下结构：

```markdown
<details>
<summary><b>数据名称</b> · 简短描述</summary>

<!-- 可选：重要提示 -->
***重要说明（如有）***

---

**子类型名称** · 可选描述

![徽章1](...)
![徽章2](...)
![徽章3](...)
![来源](...)

🔗 [链接名称](URL) · 📅 数据时段 · 📝 [示例](./path) · ⚠️ 注意事项

---

**另一个子类型**

...

</details>
```

### 链接图标说明

| 图标 | 含义 | 示例 |
|------|------|------|
| 🔗 | 数据链接 | 🔗 [GFS_NOAA](https://...) |
| 📅 | 数据时段 | 📅 最近 10 日 |
| 📝 | 示例代码/文档 | 📝 [Python示例](./sources/example.py) |
| ⚠️ | 注意事项 | ⚠️ 需登录 |
| 🔄 | 数据状态 | 🔄 近实时 |
| 🪜 | 需特殊网络 | 🪜 需VPN访问 |
| 🔍 | 搜索提示 | 🔍 搜索关键词 |

### 数据字段要求

| 字段 | 必需 | 描述 | 示例 |
|------|------|------|------|
| 数据名称 | 是 | 数据产品或服务的名称 | GFS 全球预报系统 |
| 机构 | 是 | 数据提供机构（在summary中） | NOAA/NCEP |
| 分辨率 | 是 | 空间分辨率（徽章） | 0.25° |
| 时间信息 | 是 | 时效或时间分辨率（徽章） | 0~384h(3h) |
| 更新频率 | 是 | 数据更新频率（徽章） | 每日4次 |
| 来源 | 是 | 数据提供方（徽章） | NOAA |
| 链接 | 是 | 数据获取链接 | [NOMADS](https://...) |
| 数据时段 | 推荐 | 历史数据覆盖范围 | 📅 2021年至今 |

## 评审流程

### Pull Request 检查清单

在提交 PR 前，请确认：

- [ ] 代码/文档遵循项目规范
- [ ] 所有链接有效且可访问
- [ ] 数据源使用折叠卡片格式（`<details>`）
- [ ] 包含所有必需徽章（分辨率、时效、更新、来源）
- [ ] 来源徽章使用正确的颜色代码
- [ ] 添加了必要的测试（如适用）
- [ ] 更新了相关文档
- [ ] 提交信息清晰明确
- [ ] 分支与主分支无冲突

### 评审标准

1. **功能性**：贡献解决了什么问题
2. **质量**：代码/文档质量是否符合标准
3. **测试**：是否包含适当的测试用例
4. **文档**：是否更新了相关文档
5. **兼容性**：是否与现有内容兼容
